## Context

The project already has `scripts/seed_report_types.py` which bootstraps report types via HTTP API.
Users with non-citizen roles (admin, agent) cannot be created via `POST /auth/register` because that
endpoint defaults all new accounts to the `citizen` role. A direct-DB seed script is therefore the
appropriate approach: it uses the same `PasswordService` (bcrypt) and `SQLAlchemyUserRepository`
that the application relies on, so passwords are hashed identically and the records are fully
compatible with the JWT auth flow.

Seed users created:

| username / email prefix | email                   | role    | password   |
|-------------------------|-------------------------|---------|------------|
| admin                   | admin@gavea.br          | admin   | admin      |
| citizen01               | citizen01@gavea.br      | citizen | citizen01  |
| agente                  | agente@gavea.br         | agent   | agente     |

The script is idempotent: if the email already exists it prints a "skipped" message and continues.

## Steps

### Step 1: Create `scripts/seed_users.py`

Create a standalone seed script that inserts the three default dev users directly into the SQLite
database using the application's own infrastructure (PasswordService + SQLAlchemyUserRepository +
SessionLocal). The script must:

1. Call `create_tables()` so it works against a fresh (empty) database.
2. Define the three users as a constant list of `(email, name, role, password)` tuples:
   - `("admin@gavea.br", "admin", UserRole.admin, "admin")`
   - `("citizen01@gavea.br", "citizen01", UserRole.citizen, "citizen01")`
   - `("agente@gavea.br", "agente", UserRole.agent, "agente")`
3. For each user: call `user_repo.find_by_email(email)` — skip with a "skipped" log line if found;
   otherwise hash the password with `PasswordService().hash_password(password)`, call
   `User.create(email, hashed, name, role)`, then `user_repo.save(user)`.
4. Print a summary `"Done. Created: N, Skipped: N"`.
5. Read `DATABASE_URL` from `os.environ` with fallback `"sqlite:///./fala_gavea.db"` (same as the
   app's own `session.py`).
6. Expose a `main()` function and a `if __name__ == "__main__": main()` guard.
7. Add a module-level docstring matching the style of `seed_report_types.py`:

   ```
   Usage:
       uv run python scripts/seed_users.py
   ```

**Architecture note**: the script bypasses the `RegisterUser` use case intentionally. `RegisterUser`
hard-codes the citizen role and validates name length ≥ 2 chars — both are fine for citizens but
incompatible with seeding admin/agent roles. Direct repository usage is the correct pattern for
infrastructure-layer seed scripts.
